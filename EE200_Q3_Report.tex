\subsection{Objective}
Build a Shazam-style audio fingerprinting system and deploy it as an interactive web application (``Zapptain America'').

\subsection{Q3A: Theory and Experiments}

\subsubsection{Experiment 1: The Spectrogram and Window Length Trade-off}

The fundamental tool for audio analysis is the Short-Time Fourier Transform (STFT). We must choose a window length (\texttt{n\_fft}).
\begin{itemize}[leftmargin=2em]
    \item A \textbf{short window} provides excellent time resolution (we know exactly \textit{when} a note occurred) but poor frequency resolution (the frequencies blur together).
    \item A \textbf{long window} provides excellent frequency resolution (we know exactly \textit{which} note was played) but poor time resolution (we don't know exactly when it started).
\end{itemize}

\begin{math_box}[Time-Frequency Uncertainty]
For STFT with window size $N$ and hop $H$ at sample rate $f_s$:
\begin{align}
    \Delta t &= H/f_s \quad\text{(time resolution)}, \quad \Delta f = f_s/N \quad\text{(frequency resolution)} \\
    &\text{Uncertainty: } \Delta t \cdot \Delta f \geq 1/(4\pi)
\end{align}
Larger $N$ gives better frequency resolution but worse time resolution. There is no free lunch.
\end{math_box}

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{q3a_experiment1.png}
    \caption{STFT spectrograms of a 5-second clip from \textit{Let It Be}. \textbf{Left} ($N=256$): crisp note onsets, blurred frequencies. \textbf{Right} ($N=4096$): distinct harmonics, smeared onsets. We chose $N=2048$ as the optimal balance for the fingerprinting system.}

\end{figure}

\subsubsection{Experiment 2: Constellation Map \& Fingerprinting (Hashing)}

Matching raw audio waveforms is impossible in a noisy cafe. Even matching raw spectrograms fails if the timing is slightly off. Instead, we extract \textbf{local maxima} (peaks) from the spectrogram to form a sparse \textit{constellation map}.

\begin{insight_box}[Why pair peaks into hashes?]
If we only match single frequencies $f_1$, we get thousands of false positives (many songs have a 440Hz note). But if we pair an anchor peak $f_1$ with a target peak $f_2$ separated by time $\Delta t$, we form a hash: $(f_1, f_2, \Delta t)$.
The probability of two different songs having the exact same two frequencies separated by the exact same time delta is exponentially lower. The pair acts as a highly specific, unique identifier.
\end{insight_box}

    \caption{Spectrogram overlaid with constellation map (red dots). Local maxima capture the ``harmonic skeleton'' of the song using fewer than 1\% of all spectrogram cells.}

\subsubsection{Experiment 3: Robustness to Noise}

We added Gaussian noise to the query clip to observe its effect on the fingerprinting algorithm.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.9\textwidth]{q3a_experiment3.png}
    \caption{Constellation Map with Heavy Noise (Blue dots). Original Peaks: 268, Noisy Peaks: 811.}
    \label{fig:noise_robustness}
\end{figure}

As we increase the noise, the background noise introduces spurious peaks that overtake the true musical peaks. If too many true peaks fall below the dynamically calculated amplitude threshold, the generated hashes will completely change, and recognition fails.

\subsubsection{Experiment 4: Pitch Shifting \& Time Stretching}

What happens if a DJ speeds up a track by 2\% (time-stretching and pitch-shifting it)?
The song sounds practically identical to humans. But our identifier is completely defeated. Why? Because the frequencies $f_1, f_2$ and the time delta $\Delta t$ are all absolute values.
A pitch shift changes $f_1 \rightarrow f_1'$, so the hash $(f_1, f_2, \Delta t)$ no longer matches the database at all.

\begin{result_box}[How to make it robust?]
Instead of hashing absolute frequencies, we could hash the \textbf{ratio} of frequencies: $\frac{f_2}{f_1}$. A pitch shift multiplies all frequencies by a constant factor, so the ratio $\frac{k \cdot f_2}{k \cdot f_1}$ remains invariant. For time-stretching, we could hash the ratio of time deltas between triplets of peaks.
\end{result_box}
